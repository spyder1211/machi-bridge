# まちブリッジ MVPプロトタイプ作成指示書

## このドキュメントの目的

「まちブリッジ」のMVPプロトタイプを構築する。目的は、企業・自治体・議員の三者がそれぞれのダッシュボードでどのような情報・価値を得られるかを体験してもらうこと。全機能の実装ではなく、「このプラットフォームがあると何が嬉しいか」を30秒で伝えられるデモを作る。

---

## 1. プラットフォーム概要

### まちブリッジとは

自治体と企業のマッチングを促進するプラットフォーム。一般社団法人ふるさと未来創造コンソーシアムが仲介役となり、自治体が抱える地域課題に対して、企業のサービス・技術をマッチングさせる。

### 三者にとっての価値

| ロール | まちブリッジで得られる価値 |
|--------|--------------------------|
| **企業** | 自治体の課題・予算が見えるから、「刺さる提案」ができる |
| **自治体** | 自分たちの課題にマッチした企業サービスが見つかり、KPI達成に近づける |
| **議員** | 自治体の課題と解決策の選択肢が整理されるから、予算案の根拠が明確になる |

この三者の体験をダッシュボードで表現するのがこのMVPの目標。

---

## 2. 技術方針

### 最優先事項: シンプルさ

- **単一のHTMLベースのReactアプリ**（Vite + React）
- **CSS**: Tailwind CSS（CDN）
- **データ**: アプリ内のJavaScript定数（外部ファイル不要）
- **ルーティング**: React内の状態切替（react-routerは不要。タブ/ボタンで画面を切り替える）
- **ビルドツール**: Vite（最小構成）
- **言語**: JavaScript（TypeScript不要）

### やらないこと

- ユーザー認証・登録フロー
- フォームのバリデーション・送信処理
- API通信・DB接続
- 管理画面
- レスポンシブ対応（デスクトップのみで十分）

---

## 3. アプリの構造

### 画面構成

アプリは1つのシェルの中で、ロール切替によってダッシュボードが切り替わるシンプルな構造にする。

```
┌─────────────────────────────────────────────────┐
│  ヘッダー: ロゴ「まちブリッジ」  [企業|自治体|議員] ロール切替タブ  │
├─────────────────────────────────────────────────┤
│                                                 │
│  選択中ロールのダッシュボード                        │
│                                                 │
└─────────────────────────────────────────────────┘
```

- ヘッダー上部にロール切替タブ（3つ）を常時表示
- タブを切り替えると、同じ自治体データを「企業目線」「自治体目線」「議員目線」で見せる
- 切替時にスムーズなトランジションを入れる

### ロール切替タブのデザイン

- 3つのタブ: 🏢 企業ダッシュボード ／ 🏛 自治体ダッシュボード ／ 🎓 議員ダッシュボード
- 選択中タブはアクティブ色＋下線
- 各タブの下に一言キャッチ:
  - 企業: 「自治体の課題から、刺さる提案を見つける」
  - 自治体: 「課題にマッチした企業サービスでKPI達成へ」
  - 議員: 「地域課題と解決策を整理して、予算案の根拠に」

## 5. ダッシュボード設計

### 5.1 企業ダッシュボード 🏢

**キャッチフレーズ**: 「自治体の課題から、刺さる提案を見つける」

**このダッシュボードの目的**: 企業が全国の自治体の課題を一覧し、自社サービスがマッチする営業先を効率的に見つけられるようにする。

#### 画面構成

**① 概要サマリー（画面上部、横並びカード3枚）**

| カード | 内容 | 表示例 |

## 6. デザイン方針

### カラーパレット

- **プライマリ**: 深緑〜ティール（#0f766e 〜 #14b8a6）— 公共・信頼のイメージ
- **背景**: 明るいグレー（#f8fafc）
- **カード背景**: 白（#ffffff）、ホバー時に薄い影
- **テキスト**: ダークグレー（#1e293b）
- **アクセント**: 各予算款・カテゴリに対応した色（モックデータ定義の通り）

### 全体のトーン

- 官公庁向けサービスとして堅実感がありつつ、データの見やすさを最優先
- KPIの進捗は視覚的に（プログレスバー、数値のハイライト）
- カード間の余白を十分に取り、情報密度を適度に保つ
- 予算款バッジ・カテゴリバッジは色で直感的に区別できるようにする

### タイポグラフィ

- フォント: Noto Sans JP（Google Fonts）
- 見出し: bold、やや大きめ
- 本文: regular、読みやすいサイズ

## 8. 注意事項

- **すべてのテキストは日本語**で記載する
- モックデータの自治体名は架空だが、課題・KPIは実際の自治体で見られるリアルな内容にしている。この具体性がデモの説得力になるのでそのまま使うこと
- 「この自治体に提案する」「オファーを送る」等のボタンはUIのみ表示し、クリック時はトースト通知（「デモ版のため、この機能は実装されていません」等）を出す
- 企業ダッシュボードでは、モック的に「株式会社デジタルガバメント」としてログインしている体でデータを表示する。自社サービスとマッチする課題を抽出してハイライトすること
- 自治体ダッシュボードは「みらい市」の担当者としてログインしている体で表示する
- 議員ダッシュボードは「みらい市議会の議員」としてログインしている体で表示する
- 「AI解析結果」であることを示す注釈（「この情報はAIにより自治体の基本計画から自動抽出されたものです」）をダッシュボード内に控えめに表示する
